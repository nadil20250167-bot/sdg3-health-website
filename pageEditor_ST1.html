HTML Structure

<header> is Represents introductory content for a page or section — typically containing the site logo, title, and main navigation. Unlike a generic <div> , it tells both browsers and assistive technology "this is the top/intro area of the content," so screen readers can let users jump straight to it.

<nav> is Wraps major navigation links (menus, tables of contents, breadcrumbs). Screen readers announce it as a "navigation landmark," letting users jump straight to menus without reading through unrelated content.

<main> is Contains the primary, unique content of the page — excludes repeated elements like headers, footers, or sidebars. There should only be one <main> per page. It helps screen readers skip directly to the core content.

<footer> is Holds closing information — copyright, contact details, related links. Usually appears at the bottom of a page or section.

Why these were chosen (structure + accessibility)

  • Better structure for developers
  • Accessibility (screen readers)
  • SEO benefits
  • Consistent browser behavior

CSS Techniques

Layout Approach: The navigation bar (.navbar) uses Flexbox (display: flex) with (justify-content: space-between) and (align-items: center). This was chosen because the navbar is a one-dimensional layout — the logo sits on one side and the navigation links on the other — and Flexbox distributes these two groups across the row automatically, while vertically centering them regardless of their height.

The footer (.site-footer) also uses Flexbox (display: flex; flex-wrap: wrap;) rather than CSS Grid. Each footer (section) is set to (flex: 1) with a (min-width), allowing columns to grow evenly and wrap onto a new row on smaller screens. Flexbox was sufficient here because the number of footer columns is small and consistent, so a full two-dimensional grid system wasn't necessary — Flexbox achieves the same responsive, evenly-spaced column layout with simpler code.

Design & Implementation Rationale

Colour Palette: The site uses a blue-based palette (#0077c0, #0b72b5) for the header, navigation bar, and section headings, paired with a soft light-green background (#F1F8F4) for the body. Blue was chosen because it conveys trust, professionalism, and calmness — fitting for a student wellbeing/university-related site — while the pale green background keeps the page feeling light and easy on the eyes rather than stark white. Accent colours on the topic tiles (green, orange, teal, navy) are used sparingly to differentiate categories without overwhelming the overall calm tone. Using the same blue consistently across the navbar, headings, and links (rather than introducing new colours per page) creates visual consistency, so users immediately recognise interactive elements (like links) no matter which page they're on.

Typography: The site uses 'Times New Roman', Times, serif as the primary font-family across the body, headings, and footer. A serif font was chosen to give the content a formal, readable, academic tone, appropriate for an educational/university context. Using one consistent font-family site-wide (rather than mixing fonts per section) ensures visual harmony and avoids the inconsistent, unpolished look that comes from multiple competing typefaces.

Spacing: Consistent padding and margin values (e.g. padding: 40px 20px; on card containers, margin-bottom: 2.5rem; on sections) are used throughout the site to create even breathing room between elements. This consistent spacing rhythm prevents content from feeling cramped or cluttered, and helps visually separate distinct sections (like the hero, cards, and topic gallery) without needing extra borders or dividers.

Usability & Consistency: Because the same colours, font, and spacing patterns are reused across every page and component (navbar, cards, footer, tables, forms), users experience a predictable, cohesive interface. This reduces cognitive load — visitors don't need to re-learn how to navigate or read content as they move between pages, since buttons, links, and headings always look and behave the same way. Consistent spacing also improves accessibility, giving elements enough clickable/tappable area and making text easier to scan and read.

HTML Structure

The mission cards are wrapped in <article> tags because each card (e.g. "Move Every Day," "Sleep Matters") is a standalone piece of content with its own heading, text, and list — it would still make sense on its own if moved elsewhere. They sit inside a <section>, which groups them together as one themed part of the page.

The topic gallery is structured similarly — a section containing clickable tiles. Each tile has a heading and short text, but since its main job is to link to another topic/page (not present standalone content), it's wrapped in an (a) tag instead of (article).

Using these semantic tags instead of plain divs makes the page easier to understand for both browsers and screen readers, and helps organise the content logically.

CSS Techniques

The card layout (.cards-container) and topic gallery (.topic-gallery) both use Flexbox (display: flex; flex-wrap: wrap;) rather than CSS Grid. This allows the cards/tiles to sit side-by-side and automatically wrap onto a new row when there isn't enough space, making the layout responsive across different screen sizes. A (gap) property adds even spacing between items without needing extra margins.

The embedded page-specific CSS includes styles like (.card), (.card-body), and (.topic-tile), which control the look of individual cards and tiles — for example, rounded corners (border-radius), shadows (box-shadow), hover animations (transform: translateY(-8px)), and different background colours for each topic tile. These styles are specific to this page's components, unlike the global styles (navbar, footer) which apply site-wide.

Design & Implementation Rationale

Cards were used to present the mission content because they break information into small, scannable chunks — each card focuses on one topic (like sleep, stress, or movement) with a heading, short description, and list of tips. This is easier for users to read than one long block of text, and lets them quickly find the topic that's relevant to them.

The interactive gallery supports navigation by turning each topic into a clickable tile that links to its own content page. Hover effects (like lifting up slightly with a shadow) give users visual feedback that the tile is clickable, making it clear this is a navigation element rather than just decoration. This makes it faster for users to explore the site, since they can jump directly to the topic they're interested in from one central gallery instead of digging through menus.

HTML Structure

The page uses a clear heading hierarchy: <h1> for the main page title (e.g. "Sleep & Mental Wellbeing"), <h2> for major section titles (e.g. "Introduction," "Tips for Better Sleep"), and <h3> for smaller sub-topics within cards or tiles (e.g. "Move Every Day"). This structure helps both users and screen readers understand the page's organisation at a glance, since headings act like a table of contents.

Internal anchor navigation is used in the "On this page" bar, where each link (e.g. #introduction, #schedule) jumps directly to its matching section using the section's id. This lets users skip straight to the part of the page they want, instead of scrolling through everything.

A fixed-position "Go to Top" link stays visible in the corner of the screen as the user scrolls down, using position: fixed. Clicking it smoothly scrolls the page back to the top, making it easier to navigate long pages without manually scrolling all the way back up.

CSS Techniques

The "Go to Top" button uses position: fixed so it stays in the same spot on the screen (e.g. bottom-right corner) no matter how far the user scrolls down the page. This is combined with values like bottom and right to pin it in place, and a high z-index to make sure it stays on top of other content rather than being hidden behind it. Clicking the button scrolls the page back to the top, which is helpful on long pages so users don't have to manually scroll all the way back up.

Images were framed using border-radius to soften their corners and box-shadow to give them a subtle lifted, card-like appearance. This makes images feel more polished and consistent with the rounded, card-based style used elsewhere on the site (like the mission cards), rather than looking like plain, flat rectangles.

Design & Implementation Rationale

Content was structured to be scannable by breaking information into short sections with clear headings, bullet-point lists, and cards rather than long paragraphs. Important terms (like "Date," "Time," or key tips) are bolded or emphasised so users can quickly spot key information without reading every word. Short paragraphs and lists make it easy for users to skim the page and find the specific topic they need (e.g. sleep, stress, nutrition) instead of reading everything from top to bottom.

Sources were chosen based on credibility and relevance to student health — prioritising reputable organisations (e.g. university health services, government health guidelines, or established health bodies) over unofficial or unverified websites. This ensures the advice given (e.g. sleep recommendations, stress management tips) is accurate, trustworthy, and appropriate for a student audience.

Accessibility Considerations

Another challenge I encountered was that my card layout wasn't behaving the way I expected — I had set a fixed height: 700px on my .card class, but the cards were displaying at different heights depending on their content, as if the fixed height wasn't being applied at all. After checking my stylesheet more closely, I realised I had accidentally defined .card twice — once earlier in the file with height: 700px, and again further down without a height set. Since CSS applies the last matching rule when there's a conflict, the second definition was silently overriding the first, so my fixed height was never actually being used.

To fix this, I removed the duplicate rule and kept only one .card definition, deciding whether or not I actually wanted a fixed height based on how the content looked.

From this, I learned to be more careful about accidentally repeating the same class name in a stylesheet, especially in longer CSS files where it's easy to forget a rule already exists further up. It also taught me that CSS follows a "last rule wins" order for conflicting properties, which is an easy thing to overlook when debugging why a style isn't applying.
