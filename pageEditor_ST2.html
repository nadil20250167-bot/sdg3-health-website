Splash Screen

Open the live page → | See validation evidence →

HTML Structure

The Splash Screen does not use the shared site template, since the specification allows it to stand apart from the header/main/footer structure as long as the look and feel still matches the rest of the site. A <meta http-equiv="refresh"> tag is placed in the <head> so the browser automatically loads home.html after four seconds, without needing any JavaScript to trigger the redirect. The visible content sits inside a single <main> element and is built from an <h1> for the site title, a <p> for the mission statement, an <h2> and <ul> listing the group members, a loading bar, a countdown message, and a skip button — using headings and lists rather than a page built entirely from generic <div> elements.

CSS Techniques

CSS custom properties declared on :root (such as --c-primary and --c-primary-dark) hold the colour palette and font family in one place, so the whole page's colour scheme can be adjusted by changing a single variable rather than editing many separate rules. The full-screen background photo uses background-size: cover together with a layered gradient overlay, which keeps the white text readable regardless of how bright or busy the underlying photo is. The site title uses clamp() so its font size scales smoothly between small and large screens instead of jumping abruptly at fixed breakpoints. The loading bar fills using a CSS @keyframes animation timed to four seconds, so it visually tracks the length of the countdown happening at the same time.

JavaScript Functionality

setInterval() updates a countdown number in a <span> once every second, giving the user a visible sense of how long is left before the redirect happens. A separate skipIntro() function is attached to the skip button; when clicked, it clears that interval and sets window.location.href to home.html, sending the user straight to the home page immediately instead of waiting out the full four seconds.

Design & Implementation Rationale

The countdown and loading bar were both kept visible at the same time because a single moving element (the bar) reinforces the number shown in the countdown text, giving the user two complementary cues that time is passing rather than relying on the number alone. The skip button was included because forcing every visitor to wait the full four seconds, even on a repeat visit, would be an unnecessary friction point — giving users control over how quickly they move past the introduction respects their time while still showing the group's branding to first-time visitors.

SDG Action Impact Simulator

Open the live page → | See validation evidence →

HTML Structure

The page uses the shared <header>, <main> and <footer> template so it fits naturally alongside the rest of the group's work. Inside <main>, the six action cards are each built from a <div class="card"> carrying a data-points attribute that stores the card's impact value, together with an image, an <h2> title, a short description and a points line. Every card also has tabindex="0", which pulls it into the normal keyboard tab order even though a <div> would not be focusable by default. The feedback panel below the cards uses an <h2>, two <span> elements for the live count and score, a badge <div>, and a reset <button>.

CSS Techniques

Embedded CSS scoped to the main container lays the cards out using display: grid with repeat(auto-fit, minmax(230px, 1fr)), which lets the number of columns adjust smoothly to the width of the screen without writing separate media queries for each breakpoint. A :focus style adds a clear outline around whichever card is currently focused, so keyboard users can always see where they are on the page. The background image behind the feedback panel is scoped only to that element, using background-size: cover and a CSS transition, so the cards above stay visually stable while only the result panel changes.

JavaScript Functionality

Every card has both a click listener and a keydown listener that checks for Enter or Space, so selecting a card behaves the same way with a mouse or a keyboard — this was required by the specification, since hover-only or CSS-only interaction was explicitly not allowed. A toggleCard() function adds or removes a selected class on click, and an updateSummary() function totals the points of every selected card, works out an impact level from fixed thresholds, and updates the score text, badge colour, feedback message and background image together in one place.

Design & Implementation Rationale

Three impact levels (low, medium, high) were chosen instead of a single pass/fail result because health habits build up gradually, and showing a spectrum of outcomes encourages users to keep adding small actions rather than feeling like they have either "succeeded" or "failed" after one choice. Changing the background image and badge colour for each level was intended to give an immediate, visual sense of progress, reinforcing the numeric score with a mood that shifts from muted to more energetic as the impact level rises.

Content Page

Open the live page → | See validation evidence →

HTML Structure

The Content Page deliberately covers a single focused sub-topic, physical activity and fitness, rather than several unrelated topics, so every section stays genuinely relevant. The page opens with an <h1> naming the topic directly, followed by a <nav class="page-nav"> containing anchor links such as href="#routine", which jump the reader straight to the matching <section id="..."> further down the page. Each section is built from an <h2>, one or more paragraphs, and at least one relevant image with descriptive alt text, keeping the heading order moving cleanly from <h1> to <h2> without skipping a level.

CSS Techniques

Embedded CSS styles the intro banner, the topic label, and the individual section cards with padding, rounded corners and soft shadows, reusing the same visual language as the AIS card layout so the two pages feel like part of one connected site. The site's shared .page-nav class from the global stylesheet is reused directly for the in-page navigation menu, rather than being redefined here, so the page-specific CSS does not duplicate or conflict with the global styles. A fixed-position <a class="go-top"> link sits in the corner of the screen at all times, using only position: fixed and an anchor link back to the page header, so returning to the top of a longer page never requires any JavaScript.

Design & Implementation Rationale

Physical activity was chosen as a single, narrow sub-topic rather than a broad overview of general wellbeing because a focused page lets each section go into genuine depth — why activity matters, what actually counts as exercise, how to build a routine, and how to work around common barriers — instead of skimming several ideas shallowly. The content is written for a general audience rather than students specifically, since fitness habits and the barriers to them (time, motivation, injury) apply just as much to working adults, parents or older readers as they do to students.

Challenges and Lessons Learned

Running each of my pages through the W3C validator early on caught several issues that were easy to miss by eye: heading levels skipping from <h1> to <h3> on the action cards, a meta refresh value missing a required space, an aria-label placed on a <div> without a role, and an 8-digit hex colour value that the validator's CSS parser did not support, which caused a confusing, misplaced error further down the file rather than pointing at the actual line at fault. The main lesson learned was to validate every page as soon as it is drafted rather than leaving it until the end, since one small issue near the top of a stylesheet can produce misleading errors elsewhere in the document, and fixing several unrelated issues at once is much harder than fixing them one at a time as they appear.

Accessibility Considerations

On the Splash Screen, the countdown text uses aria-live="polite" so screen readers announce the changing number without interrupting other content, and the loading indicator is wrapped in an element with role="group" and an aria-label describing its purpose, since aria-label is only valid on a <div> once it has been given a role. A redundant role="main" was also removed from the <main> element, since that role is already implied by the element itself and repeating it is unnecessary.

On the Action Impact Simulator, every card image includes descriptive alt text, and every card can be selected with the keyboard as well as the mouse, meeting the requirement that interactivity is not CSS-only or mouse-only. On the Content Page, all images include meaningful alt text, headings run in a logical order with no levels skipped, and the internal navigation links give an additional way to move through a longer page without relying on scrolling alone.
