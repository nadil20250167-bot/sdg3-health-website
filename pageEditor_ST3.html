Feedback & Programme Directory Page

HTML Structure

The page reuses the shared <header>, <main> and <footer> from the template. Inside <main>, a scrollable <section> holds four programme <div class="card"> elements, followed by a ratings summary section and a <form> split into two <fieldset> groups ("Personal Details" and "Feedback Details") with <legend> labels for accessibility.

CSS Techniques

The programme list uses a flexbox-based .cards-container (from the Global CSS) inside a scrollable container restricted with max-height and overflow-y. Buttons use a transition on background-color and transform for the hover effect required by the brief.

JavaScript Functionality

An input event listener updates a live character counter on the comments textarea. A submit event listener intercepts the form, checks each required field with simple validation rules, toggles per-field error messages, and reveals a confirmation message once every field passes.

Design Rationale

Fieldsets group related inputs so screen reader users hear the section purpose before each field, and the character counter gives immediate feedback on the message length limit rather than only failing silently on submit.

Team Page

HTML Structure

Four team member blocks sit inside <main> as focusable <div> elements (tabindex="0"), each containing a thumbnail image, a name, a role, and a hidden details panel.

CSS Techniques

Hover and focus states are handled with the same rule set (:hover, :focus) so mouse and keyboard users get identical feedback. The details panel is animated by transitioning max-height and opacity rather than display, which keeps the transition smooth.

Design Rationale

Adding tabindex="0" and a visible outline on :focus ensures the interactive thumbnails are usable without a mouse, addressing the coursework's accessibility requirement directly.

Challenges and Lessons Learned

The main technical challenge was making the team member details panel keyboard-accessible without JavaScript. Using :focus-within on the parent block solved this, so the panel reveals whether the user tabs onto the container or a link inside it. I also learned that character-counter inputs need to update on every keystroke, not just on blur, otherwise the feedback feels laggy and unhelpful.

Accessibility Considerations

  • All images include descriptive alt text (e.g. "Volunteer checking a person's blood pressure at a stall") rather than generic labels.
  • Form fields are grouped with fieldset/legend and each input has an associated label.
  • Interactive team thumbnails are keyboard-focusable with a visible focus outline.
  • Error messages appear directly beneath their related field so they are easy to associate with the input.
