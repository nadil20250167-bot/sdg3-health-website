Technical Description: User Profile Page

Open the live page in a new tab | See its validation evidence

HTML Structure

The page reuses the shared <header>, <nav> and <footer>. The <main> area has one <section class="card"> per step (Exercise, Sleep, Diet) plus one more for the live output box. No <div> anywhere - just <section>.

CSS Techniques

Steps reuse the shared .card class for the rounded, shadowed box look. The "Redo this step" button gets its own page-specific .secondary-btn class (dashed outline instead of a solid fill) so it doesn't get mixed up with the main Start button.

JavaScript Technical Description

Answers are collected with prompt() asked one at a time inside a for loop, not through an HTML form. Each step lives in a stepConfig object holding its 3 questions and which step comes next. Clicking "Start" calls runStep(), which loops through the 3 prompts, saves each answer (or the string "Skipped" if the box was left blank or Cancel was hit) into the profileData object, then reveals the next step by flipping its style.display from none to block. A separate updateProfile() function rebuilds the output box and completion bar every time a step finishes, using document.getElementById() and .innerHTML - no page reload involved.

Design and Implementation Rationale

I went with prompt() because that is exactly what the module spec asks for on this page, even though it feels a bit old school compared to buttons. Gating the steps (Sleep stays hidden until Exercise is done) stops the page feeling like a wall of buttons or prompts all at once, and keeps it closer to the "progressive reveal" behaviour the brief describes. Letting people skip by leaving a prompt blank means nobody gets stuck if they don't want to answer something personal, which felt like the right call for a health habits page.

Technical Description: Sitemap Page

Open the live page in a new tab | See its validation evidence

HTML Structure

Built entirely in SVG rather than a bitmap image, so every page in the site is a real, clickable <a> element wrapping a <rect> and <text> pair. <title> elements give each link an accessible name for screen readers.

CSS Techniques

Colour-coded node classes (.secondary, .tertiary, .current) show hierarchy at a glance. :hover and :focus-visible selectors change the rect's fill and stroke so both mouse and keyboard users get the same feedback.

JavaScript Technical Description

There isn't any JavaScript on this page - the brief only asks for CSS hover feedback and keyboard focus states here, both of which can be done natively with :hover, :focus-visible and tabindex="0". Adding JS would have been extra complexity for no real gain.

Design and Implementation Rationale

SVG made more sense than a screenshot or bitmap image because every box needed to actually be clickable and scale cleanly on different screen sizes - a raster image cannot do either of those well. Straight <line> connectors kept the diagram simple since that was the only line type available; curved connectors were not worth the extra complexity for a coursework sitemap. tabindex="0" plus a visible focus outline makes sure keyboard-only users can reach every node, and the viewBox attribute lets the whole thing shrink or grow without pixelating.

Technical Description: Content Page

Open the live page in a new tab | See its validation evidence

HTML Structure

Content is split into clearly headed <section> elements (What Is Stress?, Common Causes, Coping Techniques, Support Network), each with a unique id so the top-of-page internal navigation can link straight to it with anchor links.

CSS Techniques

A fixed-position "Go to Top" link (position: fixed) stays on screen while scrolling, built with HTML and CSS only - no JavaScript needed for that part.

JavaScript Technical Description

None on this page either - the brief says the "Go to Top" button can be pure HTML and CSS, and since the rest of the page is just static content and internal anchor links, there was not anything left that actually needed JavaScript.

Design and Implementation Rationale

The page is long enough that scrolling to find a specific section gets annoying fast, so the internal nav menu at the top and the "Go to Top" button both exist purely to save the reader's thumb. Images were picked to match each section literally (a messy desk for "causes", someone breathing outdoors for "breathing techniques") rather than being generic stock filler, and the heading levels stay strictly h1 to h2 to h3 with no skipped levels so the page reads properly with a screen reader.

Challenges and Lessons Learned

The trickiest part was getting the completion percentage to actually match what the user had done - my first attempt counted skipped answers as answered, which made the bar hit 100% even when someone had skipped half the questions. Fixing it meant storing skipped answers as the string "Skipped" instead of just leaving them blank, so I could check for that specific value when calculating the percentage. Positioning the SVG connector lines on the sitemap was also more fiddly than expected - I ended up sketching the layout on paper first and working out each rect's centre point by hand before writing any coordinates, which made the whole thing much less trial and error.

Accessibility Considerations

  • All images include descriptive alt text.
  • Every SVG link has a <title> for its accessible name plus tabindex="0" so it is reachable by keyboard.
  • Visible :focus-visible outlines are applied to links, buttons, and SVG nodes.
  • Headings follow a logical order (h1 to h2 to h3) with no levels skipped.

None of this is just box-ticking - someone navigating with a keyboard or screen reader needs to actually be able to tell what a link does and where they currently are on the page, and a missed alt tag or skipped heading level is the kind of thing that quietly breaks that for them without anyone else noticing.

Compliance with JANET Regulations

All the content on my pages is written by me for this project, not copied from the UN SDG site or anywhere else. The stock images used on the content page are sourced from Unsplash and Pexels, which are free for this kind of use, and I have kept the source noted so they can be credited properly. There is nothing offensive, defamatory, or unrelated to the coursework anywhere on my pages, and the site is only being used to host and demonstrate university coursework - which is exactly what the university's network is meant to be used for.
